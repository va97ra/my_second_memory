# Читать первым: Моя вторая память

Этот файл нужен как рабочая память проекта. Перед новыми изменениями сначала читать его, потом `README.md`, потом `docs/architecture.md`.

Проект сейчас находится в стадии живого MVP: мы не делаем маркетинговую страницу, а собираем реальное локальное приложение для личной памяти, где основное действие происходит через календарь и дневные записи.

## Что это за приложение

`Моя вторая память` - локальное Flutter-приложение для сохранения личных записей по датам. Главная идея: пользователь фиксирует события, мысли, задачи, покупки, цели, документы, места, фото и голосовые заметки в календаре, а приложение показывает нужное в ленте дня.

Приложение должно ощущаться как личная память по датам, а не как CRM или список форм. Календарный день показывает записи выбранной даты и даёт одну понятную кнопку добавления.

## Главные продуктовые правила

- На главном экране есть только три вкладки: `Лента`, `Календарь`, `Настройки`.
- В ленте нельзя создавать записи. Лента только показывает информацию.
- Создание записей идет через календарь: нажать дату, открыть день, нажать `Добавить запись`.
- В календарном дне не должно быть нижнего поля ввода, микрофона и камеры.
- Кнопка `Добавить запись` открывает редактор новой записи с уже выбранной датой.
- Фото, голос, текст и тип записи добавляются внутри редактора.
- Полноэкранное редактирование открывается по нажатию на запись в дне календаря.
- Карточка в ленте открывает только безопасный просмотр записи, без редактирования.
- Редактирование записи открывается из календарного дня и остается рабочим сценарием календаря.
- Фото в ленте, просмотре записи, календарном дне и редакторе можно открыть на весь экран нажатием на миниатюру.
- Выполнение через зеленую галочку работает только на карточках ленты.
- Лента показывает все неархивные записи календаря, сгруппированные по датам.
- Архивирование из ленты скрывает запись только из ленты; в календаре запись остаётся как история даты.
- `База памяти` из настроек - это архив скрытых из ленты записей, откуда их можно вернуть.
- В редакторе записи не должно быть переключателя `Выполнено`.
- Удаление не показывается на карточках ленты и не засоряет чат; удаление остается в меню полноэкранного редактора.

## Текущее состояние интерфейса

### Лента

Экран `Лента дня` показывает все неархивные записи, сгруппированные по датам. Новые даты идут выше старых.

Важно: пустые заголовки дней не должны висеть в ленте. День появляется только если в нём есть неархивные записи.

Карточка ленты сейчас:

- показывает только иконку типа, без текстового слова типа;
- не показывает число и месяц внутри карточки, потому что дата стоит над группой записей;
- не дублирует тело записи, если оно уже стало заголовком;
- имеет серую галочку, пока запись не выполнена;
- после выполнения получает зеленую галочку, зеленоватый фон и остается в ленте;
- открывает полноэкранный просмотр записи по нажатию на карточку.
- не открывает редактор, чтобы случайно не повредить запись.
- имеет отдельную кнопку архивации, которая скрывает запись из ленты.

### База памяти

`База памяти` открывается из настроек по маршруту `/memory`.

Это не отдельная вкладка нижней навигации и не дубль ленты. Сейчас это архив:

- показывает только записи со статусом `archived`;
- активные и выполненные записи в базе памяти не показываются;
- карточка архива открывает безопасный просмотр записи;
- у архивной записи есть действие `Вернуть в ленту`;
- восстановление переводит запись в `active`;
- удаление по-прежнему остаётся только в редакторе записи.

### Календарь

Календарь сделан в более живом Fluent-стиле:

- локальный шрифт Manrope;
- цветная шапка месяца;
- заметные кнопки переключения месяца;
- кнопка `Сегодня`;
- цветные плитки дней;
- маркеры и счетчики записей на датах.

Нажатие на дату открывает маршрут:

```text
/calendar/day?date=YYYY-MM-DD
```

### День календаря

День календаря:

- сверху дата дня;
- в центре компактные цветные карточки/пузырьки записей;
- снизу одна кнопка `Добавить запись`;
- нажатие на существующую запись открывает редактор;
- фото и голос сохраняются как часть записи через редактор.

Кнопки в редакторе должны быть одинаковыми по смыслу:

- камера = фото;
- микрофон = голос.

Не использовать скрепку для фото.

### Редактор записи

Редактор открывается по маршруту:

```text
/memory/item/:id
```

Новая запись из календарного дня открывается по маршруту:

```text
/memory/new?date=YYYY-MM-DD
```

Просмотр записи из ленты открывается по маршруту:

```text
/memory/view/:id
```

Экран просмотра записи:

- только показывает данные;
- не содержит сохранения, удаления, выбора типа и полей ввода;
- показывает дату, тип, текст, голос и фото;
- фото открываются на весь экран по нажатию.

Текущий вид редактора:

- AppBar с кнопкой назад, заголовком `Редактировать запись`, датой под заголовком, кнопкой сохранения и меню удаления.
- Дата находится сверху под заголовком, без слова `Дата` и без иконки календаря. По нажатию на дату открывается выбор даты.
- Блока `Название` нет. Заголовок формируется автоматически из первых строк поля `Запись`.
- Блок `Тип записи` выглядит как белая панель в стиле блока `Запись`.
- Тип выбирается через всплывающее окно.
- В пользовательском выборе типов нет `Голос`, `Человек`, `Привычка`.
- Блок `Запись` занимает весь оставшийся экран и адаптируется под высоту.
- Кнопки фото и микрофона находятся справа снизу внутри блока `Запись`, под правую руку.
- Фото показываются сверху внутри блока записи, над текстом.
- Голос можно записать и сохранить; существующий голос можно прослушать.
- Голосовая запись должна показывать подпись с датой записи и длительностью.
- Переключателя `Выполнено` в редакторе нет.

## Модель данных

Основная модель: `MemoryItem`.

Важные поля:

- `id`
- `type`
- `title`
- `body`
- `memoryDate`
- `createdAt`
- `updatedAt`
- `status`
- `audioPath`
- `audioDurationSeconds`
- `imagePaths`

`memoryDate` обязателен. Он определяет, где запись появляется в календаре и ленте.

`status` сейчас используется для:

- `active`
- `done`
- `archived`

`isDone` влияет на карточки ленты.

## Навигация

Основные маршруты:

```text
/                    Лента
/calendar            Календарь
/calendar/day        Чат выбранного дня
/settings            Настройки
/settings/shifts     Графики смен
/security            PIN и биометрия
/memory/item/:id     Полноэкранный редактор записи
/memory/view/:id     Безопасный просмотр записи

```

Маршрут `/memory` открывает архивную `Базу памяти`.

В основной нижней навигации должны оставаться только:

- `Лента`
- `Календарь`
- `Настройки`

## Хранилище

Сейчас активное хранилище записей на Android/Windows/macOS/Linux/iOS - SQLite через Drift (`SqliteMemoryRepository`).

На Web временно остается `LocalMemoryRepository` через `SharedPreferences`, чтобы браузерная проверка не ломалась до отдельной SQLite Web-настройки.

Пустое хранилище должно оставаться пустым. Нельзя автоматически подмешивать демонстрационные или тестовые записи в рабочее приложение.

UI работает через интерфейс `MemoryRepository`, поэтому экраны не должны знать, SQLite сейчас используется или Web fallback.

При первом запуске SQLite-репозиторий мигрирует старые записи из `SharedPreferences` ключа `memory_items_v1`, убирает старые демонстрационные записи `starter-event`, `starter-project`, `starter-person` и ставит флаг `memory_items_sqlite_migrated_v1`.

Графики смен хранятся отдельно от записей памяти через `LocalShiftScheduleRepository` в `SharedPreferences`.

Модель графика смен: `ShiftSchedule`.

Поля:

- `id`
- `organizationName`
- `colorValue`
- `startDate`
- `workDays`
- `restDays`
- `isEnabled`

Расчёт графика простой и циклический: начиная с `startDate`, повторяется блок `workDays` рабочих дней и `restDays` выходных дней. Схема `сутки/трое` хранится как `1/3`, поэтому красится только день начала суточной смены.

Фото:

- на Web сохраняются как data URL;
- на обычных платформах выбранное фото копируется во внутреннюю папку приложения;
- в `MemoryItem.imagePaths` сохраняется стабильный локальный путь к копии фото, а не временный путь системного picker.

Голос:

- записывается локально через `record`;
- путь хранится в `audioPath`;
- длительность хранится в `audioDurationSeconds`.
- в интерфейсе голос показывается как `Голосовое сообщение` с датой записи и длительностью.

## Текущий визуальный стиль

Выбран стиль: живой Fluent / Windows 11, но не строго системный WinUI.

Уже сделано:

- Manrope подключен локально;
- светло-серый фон;
- белые панели;
- тонкие рамки;
- синий акцент;
- цветные типы записей;
- компактные карточки;
- 8px радиусы;
- без больших маркетинговых блоков.

Важно продолжать:

- не возвращать скучный черно-белый вид;
- не добавлять лишние подписи и объясняющий текст в интерфейс;
- не делать карточки внутри карточек;
- не делать нижнюю навигацию больше трех пунктов;
- держать элементы компактными.

```powershell
$env:PUB_CACHE=(Resolve-Path '.tools\pub-cache').Path
$env:APPDATA=(Resolve-Path '.tools\appdata').Path
$env:LOCALAPPDATA=(Resolve-Path '.tools\localappdata').Path
$env:FLUTTER_SUPPRESS_ANALYTICS='true'
.\.tools\flutter\bin\flutter.bat build apk --release
```

Готовый APK Flutter кладет сюда:

```text
build/app/outputs/flutter-apk/app-release.apk
```

Для локального запуска:

```powershell
$env:PUB_CACHE=(Resolve-Path '.tools\pub-cache').Path
$env:APPDATA=(Resolve-Path '.tools\appdata').Path
$env:LOCALAPPDATA=(Resolve-Path '.tools\localappdata').Path
$env:FLUTTER_SUPPRESS_ANALYTICS='true'
.\.tools\flutter\bin\flutter.bat run -d web-server --web-hostname 127.0.0.1 --web-port 51937
```

Локальная проверка обычно идет по адресу:

```text
http://127.0.0.1:51937/
```

## Последний смысловой ориентир

Пользователь строит приложение не как техническую демку, а как удобную личную память. Поэтому при сомнении выбирать:

- меньше лишних кнопок;
- больше пространства для самой записи;
- создание через календарь;
- лента только для просмотра;
- аккуратный цветной Fluent-стиль;
- одинаковые иконки и поведение на похожих экранах;
- быстрые проверки в браузере после каждого визуального изменения.
